# ADR-002: Выбор Embedding модели

## Статус
Принято

## Контекст

Для GraphRAG платформы необходима embedding-модель, обеспечивающая:
- Высокое качество семантических представлений для русскоязычных юридических текстов
- Поддержку длинных документов (статьи законов, постановления — от 512 до нескольких тысяч токенов)
- Эффективное использование ресурсов (CPU/RAM, минимальное потребление GPU)
- Self-hosted развёртывание без облачных API

**Контекст использования:**
- Индексация датасетов RusLawOD и RFSD (русскоязычные юридические документы)
- Семантический поиск по векторной БД (Qdrant)
- Гибридный поиск (dense + sparse) для повышения recall

**Аппаратный бюджет для embedding:**
- GPU VRAM уже занят LLM (~6.5 GB) — embedding-модель должна работать на **CPU + RAM**
- 64 GB RAM позволяет держать embedding-модель в памяти
- Ryzen 3900X (12C/24T) обеспечивает параллельную обработку batches

## Рассмотренные варианты

### 1. BAAI/bge-m3
- **Параметры**: ~568M
- **Размер**: ~2.2 GB (FP32), ~1.1 GB (FP16)
- **Контекст**: до **8192 токенов**
- **Языки**: 100+ языков, включая русский
- **Возможности**: Dense retrieval + Sparse retrieval (lexical) + ColBERT (multi-vector)
- **Качество**: Топ MTEB/MIRACL для мультиязычных задач
- **Бенчмарки (MIRACL Russian)**: recall@10 ≈ 0.85–0.90
- **Hybrid search**: встроенная поддержка sparse embeddings (BM25-like)

### 2. intfloat/multilingual-e5-large
- **Параметры**: ~560M
- **Размер**: ~2.1 GB (FP32), ~1.05 GB (FP16)
- **Контекст**: до **512 токенов**
- **Языки**: мультиязычная, хорошая поддержка русского
- **Возможности**: только dense retrieval
- **Качество**: хорошее, но уступает bge-m3 на мультиязычных бенчмарках
- **Особенность**: требует префиксы `query:` / `passage:` для оптимального качества

### 3. ai-forever/rubert-large (и варианты)
- **Параметры**: ~427M (rubert-tiny2) / ~340M (rubert-large)
- **Размер**: ~1.3 GB (FP32)
- **Контекст**: до **512 токенов**
- **Языки**: **только русский** (+ некоторые языки в tiny-вариантах)
- **Возможности**: только dense retrieval
- **Качество**: хорошее для русского, но уступает мультиязычным моделям последнего поколения на ruMTEB
- **Преимущество**: наименьший размер, самая быстрая инференция

### 4. intfloat/multilingual-e5-large-instruct
- **Параметры**: ~560M
- **Размер**: ~2.1 GB (FP32)
- **Контекст**: до **512 токенов**
- **Языки**: мультиязычная
- **Возможности**: instruction-tuned версия e5-large, немного лучше на специфических задачах
- **Ограничение**: тот же лимит контекста 512 токенов

## Решение

**Выбран: BAAI/bge-m3**

### Конфигурация развёртывания:

| Параметр | Значение |
|---|---|
| Модель | `BAAI/bge-m3` |
| Precision | FP16 (через sentence-transformers) |
| RAM | ~1.1 GB |
| Инференция | CPU (Ryzen 3900X) |
| Max sequence length | 8192 токенов |
| Embedding dimension | 1024 (dense) |
| Batch size | 32 (для CPU) |
| Retrieval modes | Dense + Sparse (гибридный) |

## Обоснование

### 1. Длинный контекст — решающее преимущество

| Модель | Max tokens | Юридический документ (~2000 слов) |
|---|---|---|
| bge-m3 | **8192** | ✅ Помещается целиком |
| e5-large | 512 | ❌ Требует chunking на ~15 частей |
| rubert | 512 | ❌ Требует chunking на ~15 частей |

Юридические документы (статьи законов, судебные решения) часто содержат 1000–3000 токенов. При контексте 512 токенов необходим агрессивный chunking, что:
- Разрывает семантические связи между частями документа
- Увеличивает количество векторов в БД в ~5–15x
- Усложняет retrieval (нужно объединять результаты из разных чанков одного документа)

bge-m3 с контекстом 8192 токенов позволяет индексировать большинство юридических документов **целиком**.

### 2. Гибридный поиск (Dense + Sparse)

bge-m3 — единственная из рассмотренных моделей, которая генерирует **одновременно**:
- **Dense embeddings** (1024-dim) — для семантического поиска
- **Sparse embeddings** (lexical) — для точного лексического поиска (аналог BM25)

Это критично для юридических текстов, где:
- Семантический поиск находит концептуально похожие нормы
- Лексический поиск находит точные ссылки (номера статей, даты, наименования)

Qdrant поддерживает гибридный поиск с обоими типами embeddings.

### 3. Качество на русском языке

На бенчмарке MIRACL (Russian):
- bge-m3: recall@10 ≈ 0.85–0.90
- e5-large: recall@10 ≈ 0.78–0.82
- rubert: нет данных на MIRACL, но уступает на ruMTEB

### 4. Ресурсоэффективность

- При FP16 модель занимает ~1.1 GB RAM — оставляет >60 GB для остальных компонентов
- CPU-инференция на Ryzen 3900X: ~50–100 документов/сек (batch 32)
- Не конкурирует с LLM за GPU VRAM

## Последствия

### Положительные:
- ✅ Контекст 8192 токенов — минимальный chunking для юридических документов
- ✅ Dense + Sparse embeddings — гибридный поиск «из коробки»
- ✅ Топовое качество на мультиязычных бенчмарках
- ✅ Широкая поддержка в sentence-transformers, LangChain, Qdrant
- ✅ Не занимает GPU VRAM — работает на CPU

### Отрицательные:
- ⚠️ Больший размер (~1.1 GB FP16) по сравнению с rubert-tiny2 (~170 MB)
- ⚠️ CPU-инференция медленнее GPU (~50–100 doc/s vs ~500+ doc/s на GPU)
- ⚠️ Embedding dimension 1024 — больше места в vector DB по сравнению с 768-dim моделями
- ⚠️ Не специализирована на юридическом русском (общая мультиязычная модель)

### Митигация:
- Индексация — batch-процесс, скорость CPU-инференции приемлема (~100K документов за ~20 мин)
- При необходимости ускорения — временная выгрузка LLM и использование GPU для batch-индексации
- 1024-dim при scalar quantization в Qdrant → ~512 bytes/vector — управляемый объём
- Для юридической специализации — возможно дообучение на RusLawOD (fine-tuning embedding)
