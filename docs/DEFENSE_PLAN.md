# План защиты выпускного проекта — GraphRAG Platform

**Тема:** Суверенная платформа интеллектуального анализа корпоративных знаний на основе гибридного GraphRAG
**Домен:** Юридический документооборот (законодательство РФ)
**Ключевая особенность:** Многоуровневый ролевой доступ (RBAC) на уровне узлов графа и векторных индексов

---

## 1. Введение и актуальность (2 мин)

**Проблема:** Современные корпоративные системы анализа знаний либо завязаны на облачные API (OpenAI, Anthropic), что недопустимо для юридического домена по требованиям конфиденциальности, либо используют линейные RAG-пайплайны, теряющие семантику связей между документами.

**Цель проекта:** Создать полностью автономную (air-gapped) платформу, способную:
- Анализировать корпоративные документы с сохранением связей между сущностями
- Обеспечить многоуровневый ролевой доступ на уровне самих данных (СУБД)
- Работать полностью на локальных моделях без внешних API
- Предоставить enterprise-уровень observability

**Актуальность:** 152-ФЗ, 98-ФЗ и иные нормативные акты РФ требуют изоляции персональных данных. Платформа демонстрирует, как построить production-ready GraphRAG в полностью суверенном контуре.

## 2. Архитектура платформы (3 мин)

### 2.1. Многоуровневая архитектура (C4 Model)

Архитектура разделена на **плоскость управления** и **плоскость данных**:

**Плоскость управления:**
- **FastAPI** — REST API (13 эндпоинтов), JWT-аутентификация, middleware (CORS, Security Headers, Logging, Metrics)
- **LangGraph** — конечный автомат (state machine) для оркестрации RAG-пайплайна: `classify → retrieve → generate → guardrails`

**Плоскость данных (4 независимых хранилища):**
- **PostgreSQL** — пользователи, сессии, чат-история, LangGraph checkpoints
- **Neo4j** — граф знаний: `Document → Chunk → Entity` со связями `MENTIONS`, `RELATES_TO`
- **Qdrant** — векторные эмбеддинги 1024-dim (bge-m3) с payload-фильтрацией по clearance_level и department
- **MinIO** — S3-совместимое хранилище: оригиналы документов + извлечённый текст

### 2.2. Инфраструктурная схема

```
Пользователь → Nginx (:3000) → FastAPI (:8000) → LangGraph Agent
                                                        ├── Qdrant (векторный поиск + RBAC-фильтр)
                                                        ├── Neo4j (графовая окрестность + Cypher RBAC)
                                                        ├── Ollama (LLM: qwen2.5:7b + Embedding: bge-m3)
                                                        ├── PostgreSQL (сессии + checkpoints)
                                                        └── MinIO (документы)

Observability: Prometheus :9090 → Grafana :3001
Tracing:      OpenTelemetry → Jaeger :16686
Dev tools:    Mailpit :8025, pgAdmin :5050, Open WebUI :3100
```

### 2.3. Сетевая сегментация (Docker Compose)

Две изолированные сети:
- **DMZ (172.20.0.0/24):** frontend, grafana
- **Internal (172.21.0.0/24):** backend, PostgreSQL, Neo4j, Qdrant, Ollama, MinIO

Внешний доступ к базам данных возможен только через бэкенд.

### 2.4. Ключевое архитектурное решение: LangGraph вместо линейных цепочек

Почему не LangChain `LLMChain`:
- **Динамическое ветвление:** На шаге классификации запроса система определяет, требует ли вопрос графового обхода (`requires_graph=true/false`)
- **Stateful контекст:** На протяжении всего жизненного цикла внутри LangGraph передаётся `AccessContext` (роль, отдел, уровень доступа) — защищённый контекст пользователя
- **Checkpointing:** Состояние диалога сохраняется в PostgreSQL через `langgraph-checkpoint-postgres`

## 3. Технологический стек и обоснование (2 мин)

| Компонент | Решение | Обоснование |
|-----------|---------|-------------|
| **LLM** | qwen2.5:7b / T-lite-it-1.0 (Q4_K_M, ~4.5 GB VRAM) | Баланс качества русского языка и экономии видеопамяти на целевом железе (RTX 4060 8 GB) |
| **Embedding** | bge-m3 (1024-dim, CPU inference) | 8192 токенов контекста, не конкурирует с LLM за GPU VRAM |
| **LLM Serving** | Ollama | Windows native, overhead 0.5 GB (vs 1.5 GB у vLLM) |
| **Vector DB** | Qdrant | Rust-based, payload filtering для RBAC прямо в HNSW-индексе |
| **Graph DB** | Neo4j 5 Community | Cypher для юридических traversal-запросов, 25+ лет зрелости |
| **Orchestration** | LangGraph | Циклы, checkpointing, state machine |
| **Object Storage** | MinIO | S3-совместимое, двухэтапное сохранение (оригинал + текст) |
| **Migrations** | Alembic | Детерминированное управление схемой, безопасное при concurrent-деплоях |
| **Auth** | JWT + bcrypt | Токены с ролевыми claims, 30-дневный expiry |

**VRAM Budget (RTX 4060 — 8 GB):**
- Веса модели Q4_K_M: 4.5 GB
- KV-cache (ctx 4096): 1.5 GB
- Ollama overhead: 0.5 GB
- Резерв: 1.5 GB

**Почему не облачные API:** Требование полной автономности (air-gapped контур). Система разворачивается на On-Premise сервере. Ни один запрос не уходит за пределы контура.

## 4. GraphRAG пайплайн: от документа к ответу (3 мин)

### Этап 1: Импорт и парсинг (Document Ingestion)

```
PDF/DOCX/TXT/MD/ZIP → S3 (MinIO) одновременно с парсингом
```

- **Форматы:** PDF (PyMuPDF), DOCX (python-docx), TXT, MD, ZIP (рекурсивная распаковка)
- **Сохранение в MinIO:** оригинал + извлечённый текст — резервный источник для генерации ответов
- **Дедупликация:** SHA-256 хеширование содержимого + проверка по имени/размеру файла в `filemetadata`

### Этап 2: Чанкинг и экстракция сущностей

- **Разбиение:** По границам предложений, 512 токенов с перекрытием 64 — предотвращает потерю контекста на стыках
- **Извлечение сущностей:** Rule-based NER для российского юридического домена:
  - Типы: `LAW` (закон), `ARTICLE` (статья), `ORGANIZATION` (организация), `COURT` (суд), `DOCUMENT` (документ), `PERSON` (лицо)
  - По умолчанию используется регулярный поиск по маскам законодательства (надёжнее LLM-экстракции для бюрократического языка)
  - Опционально: LLM-экстракция через локальную модель

### Этап 3: Построение графа знаний (Neo4j)

- **Детерминированная идентификация:** Entity ID = SHA-256(name + type) — исключает дублирование узлов
- **Модель графа:**
  - `Document` — исходный документ с метаданными и RBAC-атрибутами
  - `Chunk` — текстовый фрагмент с позицией, clearance_level, department
  - `Entity` — извлечённая сущность
  - Связи: `CONTAINS`, `MENTIONS`, `RELATES_TO`, `APPEARS_IN`
- **Cypher RBAC:** При запросе окрестности сущности генерируется WHERE-фильтр:
  ```cypher
  MATCH (e:Entity {name: $name})-[r]-(n)
  WHERE n.clearance_level <= $user_clearance
    AND n.department = $user_department
  RETURN e, r, n
  ```
  Фильтрация на уровне СУБД — недоступные узлы физически не возвращаются.

### Этап 4: Векторное индексирование (Qdrant)

- **Генерация эмбеддингов:** Пакетная через Ollama API (один POST с массивом строк до 64 элементов)
- **Payload:** `{chunk_id, document_id, text, position, clearance_level, department}`
- **Point ID:** UUID5 из chunk_id (детерминированный)
- **Поиск с RBAC:**
  ```json
  {
    "vector": [...],
    "filter": {
      "must": [
        {"key": "clearance_level", "range": {"lte": 2}},
        {"key": "department", "match": {"value": "legal"}}
      ]
    }
  }
  ```
  Фильтрация внутри HNSW-графа — недоступные чанки не участвуют в поиске.

### Шаг в LangGraph: classify → retrieve → generate

1. **classify:** Оценка характера вопроса → `requires_graph: true/false`
2. **retrieve:** Параллельный поиск в Qdrant (векторный) и Neo4j (графовый) с RBAC-фильтрацией
3. **generate:** Сборка контекста из найденных чанков + графовых связей → LLM
4. **guardrails:** Фильтрация выхода модели (PII, injection) перед отправкой клиенту

## 5. Многоуровневая безопасность (2 мин)

### 5.1. Ролевой доступ (RBAC) — трёхкомпонентная модель

| Компонент | Значения | Применение |
|-----------|----------|------------|
| **Роль (role)** | `admin`, `analyst`, `viewer` | Права на эндпоинты (middleware) |
| **Отдел (department)** | `all`, `legal`, `research`, `management`, `compliance`, `hr`, `finance`, `it` | Фильтрация документов по принадлежности |
| **Уровень доступа (clearance_level)** | 0=public, 1=internal, 2=confidential, 3=secret | Вертикальная иерархия доступа |

**Принцип минимальных привилегий:**
- Viewer видит только public-документы (level 0)
- Analyst видит public + internal своего отдела (level 0–1)
- Admin имеет байпас-правило — отключает все фильтры

**Применение на трёх уровнях:**
1. **API middleware:** Проверка роли перед вызовом эндпоинта
2. **Neo4j Cypher:** WHERE-условия в запросах графа
3. **Qdrant payload:** Фильтрация внутри векторного поиска

### 5.2. Guardrails — защита контента

**Двухуровневая PII-фильтрация (новое):**
- **Уровень 1:** Точные регулярные выражения (каноничные форматы: ИНН, СНИЛС, паспорт, телефон)
- **Уровень 2:** Нормализация пробелов и повторный поиск — предотвращает обход через `"1 2 3 4 5 6 7 8 9 0"` вместо `"1234567890"`

**Защита от prompt injection:**
- 16 паттернов на русском и английском языках
- Системные промпты защищены делимитерами: ```system, [SYSTEM], <|im_start|>

**Двусторонний контроль:** PII маскируется как на входе (от пользователя), так и на выходе (из LLM).

### 5.3. Аутентификация и имперсонация

- JWT с claims: `sub`, `email`, `role`
- bcrypt-хеширование паролей
- **Активация аккаунтов:** Регистрация → email → админ активирует → email об активации
- **Имперсонация:** Администратор может войти под любым пользователем с сохранением оригинального токена для возврата

## 6. API и интеграция баз данных (1 мин)

### 6.1. REST API (FastAPI, 13 групп эндпоинтов)

| Группа | Ключевые эндпоинты |
|--------|-------------------|
| **Auth** | register, login, JWT verify, sessions CRUD, users CRUD, impersonate |
| **Chat** | `/chat` (JSON), `/chat/stream` (SSE), history, delete |
| **Ingest** | text, file (PDF/DOCX/TXT/MD/ZIP), url, status polling |
| **Graph** | visualize (RBAC), search, entity, stats, documents CRUD, download |
| **Departments** | CRUD с проверкой наличия пользователей |
| **Admin** | users pagination with filtering, activation/deactivation |
| **Tests** | запуск pytest через SSE |
| **Health** | `/health` — статус Ollama, Neo4j, Qdrant |

### 6.2. Взаимодействие с базами данных

- **PostgreSQL:** Через SQLModel (синхронный Engine с QueuePool, пул 20 + overflow 10)
- **Neo4j:** Асинхронный драйвер, connection pool 50, уникальные constraints и индексы по полям безопасности
- **Qdrant:** Явные типы индексирования полей метаданных для ускорения фильтрации
- **MinIO:** Двухфазное сохранение (оригинал → S3, текст → S3), резервный источник при генерации
- **Alembic:** Программный вызов `command.upgrade("head")` при старте приложения, с fallback на `SQLModel.metadata.create_all`

## 9. Реализованные улучшения по итогам рецензии (1 мин)

После архитектурной рецензии внедрены 5 ключевых доработок:

1. **Alembic для миграций БД (Fix #1):**
   - Убран прямой `SQLModel.metadata.create_all` и сырые SQL-запросы
   - Внедрён Alembic с автогенерацией миграций и транзакционным DDL
   - Миграции запускаются при старте приложения (без гонки за структуру)
   - Fallback на `create_all`, если Alembic недоступен

2. **Специфичная обработка ошибок уникальности (Fix #2):**
   - В seed-функции `_seed_demo_users()` широкий `except Exception` заменён на точечный перехват `HTTPException(409)` + `IntegrityError`
   - Это исключает маскирование ошибок подключения к БД под race condition

3. **Пакетная генерация эмбеддингов (Fix #3):**
   - Вместо цикла `for text in texts: POST /api/embed` используется один запрос с массивом `"input": [text1, text2, ...]`
   - Батчи по 64 текста — сокращение сетевых накладных расходов в 64 раза

4. **Улучшенная PII-фильтрация (Fix #4):**
   - Двухуровневый детектор: точные regex + нормализованные regex
   - Второй проход ловит обход через пробелы/дефисы между цифрами
   - Метод `_map_normalised_span_to_raw` для точной замены в исходном тексте

5. **Асинхронная загрузка файлов (Fix #5):**
   - Синхронные `open/write/fsync` заменены на `await asyncio.to_thread()`
   - Event loop больше не блокируется при загрузке больших ZIP-архивов
